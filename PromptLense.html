How It Works
Annotation Types
  • High-impact — format specs, numeric values, tech/language names. These tokens most directly shape the model's output structure and content.
  • Constraint — words like must, only, never, no more than. Hard limits the model is instructed not to violate.
  • Context — background info, environment details, audience. Useful signal that helps the model reason about scope and framing.
  • Vague — imprecise qualifiers like good, fast, simple, soon. These match against common patterns per goal and are flagged as under-specified.
  • Filler — courtesy phrases (please, could you, thanks) and hedge words (basically, sort of). These consume tokens without improving model behavior.
Priority & Overlap

Each rule has a priority (1–5). When patterns from different rules match the same word, the higher-priority rule wins. All rules for the selected goal are applied; results are merged by scanning each character position and keeping the highest-priority annotation.

Scoring Formula

Word counts per annotation type feed a weighted formula:

positive = (high×2 + constraint×2 + context×1.2) ÷ words
negative = (filler×1.5 + vague×1.0) ÷ words
score = 40 + positive×80 − negative×60

The baseline of 40 means a plain, unannotated prompt starts at 40/100. Specificity and constraints push it up; filler and vagueness pull it down. The result is clamped to 0–100.

Score Thresholds
  • 85–100 Excellent — dense, specific, well-constrained
  • 70–84 Strong — good signal, minor refinements possible
  • 50–69 Moderate — solid base, reduce filler or add constraints
  • 30–49 Weak — too vague or too much filler
  • 0–29 Low — consider removing filler and adding specifics
Goal Tuning

Each goal loads a different rule set. Switching goals re-runs all pattern matching, changes which Missing Elements are checked, and swaps the Suggestions pool — so the same prompt can score very differently depending on what you're optimizing for.
